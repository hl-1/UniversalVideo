# VideoDream 需求分析

## 1. 项目目标

VideoDream 是一个轻量视频下载工作台，面向需要整理公开视频、学习留档、素材归档的用户。首版目标是用尽量简单的方式跑通核心流程：输入公开视频链接，解析视频信息，创建下载任务，查看状态，下载本地文件。

## 2. 用户与场景

- 内容运营：保存公开视频素材，便于后续剪辑、归档和复盘。
- 学习用户：保存公开课、公开教程等允许留存的视频。
- 个人资料整理：把分散在公开视频平台的资料统一保存到本机。

## 3. 首版范围

- 支持用户粘贴视频链接。
- 后端基于 yt-dlp 解析公开视频信息。
- 展示标题、封面、作者、时长、可选格式。
- 用户选择格式后创建下载任务。
- 页面轮询任务状态并展示进度、失败原因和下载入口。
- 下载文件保存在项目根目录 `downloads/`。
- 不接数据库，不做用户系统，不做支付系统。

## 4. 明确不做

- 不做平台登录托管。
- 不做会员、私密、付费、受限内容下载。
- 不做规避平台限制的能力。
- 不保证所有平台都可用，只承诺使用 yt-dlp 通用能力。
- 不做长期文件库，重启后内存任务状态会丢失。

## 5. UI 与体验要求

- 参考 `https://ai.codefather.cn/painting` 的风格：白底、轻导航、蓝色高亮、胶囊输入框、柔和阴影卡片、模板化高转化文案。
- 首屏就是下载工作台，不做纯营销页。
- 文案要有付费工具感，但必须包含版权和授权提醒。
- 页面必须适配桌面和移动端，避免文字溢出和控件重叠。

## 6. 验收标准

- 前端服务可访问：`http://127.0.0.1:5174`。
- 后端服务可访问：`http://127.0.0.1:8000/api/health` 返回 `ok`。
- 本地测试视频可完成解析、下载、任务查询、文件下载。
- 失败场景能展示明确错误，不让页面卡死。

## 7. 当前测试结论

- Python 3.12.10、项目 `.venv`、后端依赖已配置完成。
- ffmpeg 已安装到 `C:\softWare\environment\ffmpeg`。
- 前端 `npm.cmd run build` 已通过。
- 后端核心链路已用本地测试视频跑通：解析成功，任务状态进入 `finished`，文件接口返回 200。
- B站链接 `https://www.bilibili.com/video/BV1oxT46REE4/` 已验证封面代理正常，下载后的 MP4 包含音频和视频。
- 抖音已接入专用公开视频解析模块，并保留 yt-dlp 兜底；但测试链接 `https://www.douyin.com/jingxuan?modal_id=7652684072266845450` 当前会触发平台 `encrypt_data_miss / fresh cookies`，说明旧公共 API 方案不能保证所有抖音链接免 Cookie 成功。
- YouTube 已配置 yt-dlp 多客户端和 Node.js runtime 兜底；测试链接 `https://www.youtube.com/watch?v=kcmAnZNXjsg` 当前环境返回机器人/登录校验，需要用户授权 cookies 才能继续。

## 8. 平台能力边界

- “支持抖音和 YouTube”应理解为：优先支持可公开访问、当前平台接口允许解析的公开视频，并给出清楚失败提示。
- 不承诺绕过登录、风控、年龄验证、地区限制、会员限制或平台反自动化校验。
- 不在用户无感知的情况下读取浏览器 cookies；涉及登录态时必须由用户主动开启“本机授权模式”或手动提供 cookies。
- “本机授权模式”仅面向用户自己电脑上的本地服务使用，不适合作为公网多人下载服务。
- 新增“任意网页视频嗅探”能力，面向普通网页中的公开 `.mp4/.webm/.mov/.m3u8` 资源；不覆盖 DRM、加密授权流和复杂动态播放器。
